# Review 08 — i18n Architecture

## 1. SEO Impact (Critical)

A single URL serving two locales is **the worst SEO posture documented by Google**. Concrete consequences:

- **Googlebot crawls from US IPs with `Accept-Language: en-US`** (per [Google Search Central — Managing multi-regional sites](https://developers.google.com/search/docs/specialty/international/managing-multi-regional-sites)). With cookie default `pt`, Googlebot will receive PT-BR HTML on every crawl. EN content is effectively **invisible to Google's index**.
- **`hreflang` alternates pointing to the same URL are ignored.** Google explicitly states: *"Each language version must list itself as well as all other language versions… each URL must be unique."* Same-URL alternates fail validation in Search Console ("hreflang tags point to the same URL").
- **Indexing penalty estimate:** EN traffic from organic search ≈ **0**. PT-BR ranks normally. This isn't a "soft penalty" — it's a structural exclusion of half the content.
- **Real-world precedent:** Vercel, Linear, Stripe — all references in the spec — use **route-based** i18n (`/en`, `/fr`, `/de`). Linear migrated *away* from cookie-based locale in 2022 specifically for SEO. The Next.js team's own [i18n routing docs](https://nextjs.org/docs/app/building-your-application/routing/internationalization) recommend dynamic segment routing for any site that wants to be indexed in multiple languages.

The spec acknowledges this ("Google indexa só uma versão") but underestimates the cost: for a B2B freelancer targeting international agencies as a *secondary* audience, the EN version is the entire international funnel. Killing its discoverability kills that channel.

## 2. Implementation Cost — Honesty Check

Spec says **~80 lines**. Realistic count:

| Piece | Lines | Notes |
|---|---|---|
| `lib/i18n/dictionaries.ts` (PT + EN, with nesting + types) | 200–400 | Grows with content; type-safe key access (`DeepKeyOf<T>`) needs ~30 lines of TS gymnastics |
| `lib/i18n/getLocale.ts` (server: cookie → Accept-Language → default) | 25 |  |
| `lib/i18n/LocaleProvider.tsx` (client context) | 40 |  |
| `lib/i18n/useT.ts` (hook with dot-path resolver) | 35 |  |
| `app/actions/setLocale.ts` (server action + `revalidatePath('/', 'layout')`) | 20 |  |
| `<html lang={locale}>` sync in `app/layout.tsx` | 5 |  |
| `LocaleToggle.tsx` client component (calls action, optimistic UI) | 50 |  |
| Type generator for keys (optional but you'll want it) | 30 |  |
| Unit tests for resolver + missing-key fallback | 60 |  |
| **Realistic total** | **~465 lines** | ~6× the spec estimate |

The 80-line figure assumes no tests, no type safety, no fallback strategy, and a flat dictionary. None of those are acceptable in production.

## 2.5. Hydration Mismatch Risk

**Scenario:** Server renders with cookie `locale=en`. Between SSR and client hydration, user clears cookies in another tab, or middleware on a CDN edge mutates the cookie. Client provider reads `locale=pt` → React throws *"Hydration failed because the server rendered text didn't match the client"* on every translated string.

**Pattern to avoid it:**

1. Server component reads cookie once and **passes locale as a prop** to `<LocaleProvider initialLocale={locale}>`.
2. Provider stores `initialLocale` in `useState` and **never reads `document.cookie` during render** — only inside `useEffect` or event handlers.
3. Toggle uses a server action that calls `revalidatePath` so the next render is fully server-driven.
4. Wrap any client-only locale-dependent UI (e.g. `Intl.DateTimeFormat` with user TZ) in `suppressHydrationWarning` on the smallest possible element.

## 3. Toggle UX Edge Cases

| Case | Behavior with current spec |
|---|---|
| Toggle from Hero, scroll preserved? | **No.** Server action + `revalidatePath` causes a soft navigation; scroll resets to top unless you explicitly capture `window.scrollY` before the action and restore after. Spec doesn't mention this. |
| Share URL with EN selected | Receiver sees **PT** (their cookie default wins). Share link is meaningless. |
| `Accept-Language: en` first visit | Spec doesn't define detection — defaults to PT. International visitor sees Portuguese first impression. |
| SSG vs SSR | Cookie-dependent rendering **forces SSR on every request** (or PPR with dynamic segment). Page cannot be statically generated. Loses CDN edge cache for HTML. |

## 4. Comparison Matrix

Scores 1–5 (5 = best).

| Option | SEO | Dev cost | UX | Share-ability | hreflang correctness | Total |
|---|---|---|---|---|---|---|
| **A. Cookie + single URL** (current) | 1 | 4 | 2 | 1 | 1 | **9** |
| **B. Route-based `/pt`, `/en`** (Next 15 native) | 5 | 3 | 5 | 5 | 5 | **23** |
| **C. Subdomain `pt.` / `en.`** | 5 | 2 | 4 | 4 | 5 | **20** |
| **D. Query string `?lang=en`** | 2 | 5 | 3 | 3 | 2 | **15** |
| **E. `next-intl` + route-based** | 5 | 4 | 5 | 5 | 5 | **24** |

Notes:
- **B** = `app/[locale]/...` with `generateStaticParams`. Static generation works. ~150 LoC of plumbing.
- **C** = DNS + cert overhead, brand fragmentation. Overkill for a portfolio.
- **D** = same SSR cost as A but at least crawlable; still split-link-equity.
- **E** = library overhead (~30 KB) but gives ICU MessageFormat, plurals, type-safe keys for free; same routing as B.

## 5. Final Recommendation — Switch to **B** (route-based, hand-rolled)

For a B2B freelance portfolio with primary BR audience and secondary international agencies, the entire purpose of having an EN version is **to be found** by those agencies. Option A (current) makes EN discoverable only by people who already have the URL — which is no one, since the audience is being acquired through search. Switching to **B** (`app/[locale]/page.tsx` with `generateStaticParams(['pt', 'en'])`) costs maybe one extra day of work, preserves SSG, makes hreflang correct, makes share-links honest, and unlocks the international funnel. Skip `next-intl` (E) — the spec's hand-rolled JSON dictionary is fine; just put it behind a route segment instead of a cookie. **PT remains the default at `/`** via middleware redirect to `/pt` (or root rewrite), so BR audience UX is untouched.

## 6. If Staying with A — Five Must-Haves the Spec Misses

1. **`Accept-Language` negotiation on first visit** before defaulting to PT — server reads `headers().get('accept-language')`, parses with `@formatjs/intl-localematcher`, sets cookie on response.
2. **Scroll position preservation** across the server-action toggle — capture `scrollY` in client, restore via `useEffect` after revalidation, or use `router.refresh()` instead of full revalidate.
3. **`Vary: Cookie` and `Vary: Accept-Language` response headers** so any CDN (Vercel edge) caches per-locale instead of poisoning one locale's HTML for the other.
4. **Self-referential `hreflang="x-default"`** plus a documented decision in `robots.txt` / sitemap to declare PT as canonical, accepting EN won't be indexed.
5. **Type-safe dictionary contract** — generate `type TranslationKey = DeepKeyOf<typeof pt>` and make `t(key: TranslationKey)` reject typos at compile time; add CI check that PT and EN dictionaries have identical key sets (missing keys should fail the build, not fall back silently).
